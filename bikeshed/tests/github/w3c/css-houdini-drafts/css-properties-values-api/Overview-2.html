1. Introduction

[css-variables] defines a new <var()> function that can be used to insert the values of custom properties into other CSS property values. Where possible, this mechanism should be preferred above the computed value modification facilities of this specification.

1.1. Supported syntax strings

"<'[property-name]'>", Where [property-name] is any existing CSS property name or any string that matches the <custom-property-name> production.

Any value that parses as a value of the specified property.

The applyCtor is a JavaScript class with the following interface: 
callback interface ApplyClass {
  void apply(ElementProxy element);

2.1. The ElementProxy interface

ElementProxy objects represent the partial state of DOM objects that are available to apply hooks running in worklet global scopes. They provide the following attributes:

inputStyle

The set of styles that the apply hook has registered a dependency on.

outputStyle

The final result of running this apply hook.

pseudo

The pseudo name of the PseudoElement that this ElementProxy proxies, or null if this ElementProxy proxies an Element.

Do we need the pseudo attribute on ElementProxy for level 1? <https://github.com/tabatkins/bikeshed/issues/73>

Come up with a better name than ElementProxy. This is more of a computation context. <https://github.com/tabatkins/bikeshed/issues/74>

2.2. The ApplyDescriptor dictionary

inputProperties, of type sequence<DOMString>

The apply function is only called for elements or pseudoelements on which the listed properties all have non-initial values.

It should it be possible to access properties on the parent. <https://github.com/tabatkins/bikeshed/issues/4>

Should this be available only if explicitly requested in inputProperties? <https://github.com/tabatkins/bikeshed/issues/2>

outputProperties, of type sequence<DOMString>

This value defines the properties for which the apply function can modify the used value.

2.3. Apply classes

Apply classes provide apply hook behavior. Each apply class must provide an apply function that will be invoked when apply hooks are being processed.

2.4. The StyleWorklet interface

StyleWorklet objects provide the context within which apply hooks are invoked. Each StyleWorklet contains a name map of apply hooks, a name map of inputs, a name map of outputs, and a list of affected output properties, all of which are initially empty.

2.5. The registerApplyHook() function

The registerApplyHook(DOMString name, VoidFunction applyCtor, ApplyDescriptor config) function registers a new apply hook for processing computed style.

When registerApplyHook(name, applyCtor, config) is called, the user agent must run the following steps:

  1. If name is not a valid <ident>, throw a NotSupportedError and abort these steps.

  2. If name is a key in the name map of apply hooks, throw a NotSupportedError and abort these steps.

  3. Let outputProperties be the value of configâs outputProperties.

  4. If the outputProperties contains a property that is in the list of affected output properties, throw a NotSupportedError and abort these steps.

    This is too inflexible. Thereâs a strong use case around writing to the same native property for different elements. Maybe throw exception to window.onError in this case? <https://github.com/tabatkins/bikeshed/issues/49>

  5. If the result of IsConstructor(argument=applyCtor) is false, throw a NotSupportedError and abort these steps.

  6. Let prototype be the result of Get(O=applyCtor, P="prototype").

  7. If the result of IsCallable(argument=Get(O=prototype, P="apply")) is false, throw a NotSupportedError and abort these steps.

  8. Let applyInstance be the result of Construct(applyCtor).

  9. Add the key-value pair (name - applyInstance) to the name map of apply hooks of the StyleWorklet.

  10. Add each property in outputProperties to the list of affected output properties of the StyleWorklet.

  11. Add the key-value pair (name - outputProperties) to the name map of outputs of the StyleWorklet.

  12. Let inputProperties be the value of configâs inputProperties.

  13. Add the key-value pair (name - inputProperties) to the name map of inputs of the StyleWorklet.

This is one instance per apply hook. Do we want one instance per invocation?

2.6. Invoking apply hooks

Each time style is recomputed for an Element, each registered ApplyDescriptor/applyHook for which any of the matching inputProperties changes as a result of that recomputation is invoked. This invocation happens after any transitions or animations registered on the Element have finished applying, in the context of a StyleWorklet.

Note: apply hooks are called after transitions/animations so that custom properties can be transitioned and still have their effect apply correctly.

Implementations may memoize the result of apply callbacks relative to the complete set of inputs provided to apply (that is, the set of attributes on ElementProxy).

This invocation takes place by following these steps for each key name in the name map of apply hooks:

  1. Let inputs be the result of looking up name on the StyleWorklet's name map of inputs.

  2. Let inputStyleMap be a new StylePropertyMapReadOnly populated with only the computed values for properties listed in inputs.

  3. Let proxy be a new ElementProxy.

    Need to fill out the ElementProxy.

  4. invoke a method on a class inside a Worklet given "apply" as the methodPropertyKey and [proxy] as the arguments with the following options:

    • To create a worklet global scope the user agent will return a new StyleWorklet

    • To lookup a class instance on a worklet global scope given a workletGlobalScope the user agent will return the result of looking up name on the workletGlobalScopeâs name map of apply hooks.

    If an exception is thrown then abort these steps.

Need to deal with the output.

Note: Apply hooks run in parallel on a given Element. The output of an apply hook is never used as input to another apply hook on the same Element - instead, a snapshot of style state is taken, all apply hooks are run with subsets of the same input snapshot, then the resulting output is written into the used style.

3. Examples

3.1. Example 1: Polyfill scale, translate, rotate

<script>
["--scale-x", "--scale-y"].forEach(function(name) {
document.registerProperty({
    name: name,
    syntax: "<number>",
    inherits: false,
    initialValue: "1"
  });
});

["--translate-x", "--translate-y"].forEach(function(name) {
  document.registerProperty({
    name: name,
    syntax: "<length>",
    inherits: false,
    initialValue: "0px"
  });
});

document.registerProperty({
  name: "--rotate",
  syntax: "<angle>",
  inherits: false,
  initialValue: "0deg"
});
</script>
<style>

#myElement {
    --translate-x: 5px;
    --translate-y: 10px;
    --rotate: 10deg;
    --scale-x: 25;
    --scale-y: 25;
}

.foobar {
    --rotate: 20deg;
}
</style>

<script>
this.registerApplyHook("transform-properties", class {
    apply(el) {
      el.outputStyle.set('transform', new TransformValue(
        [
          new Translation(el.inputStyle.get('--translate-x'),
                          el.inputStyle.get('--translate-y')),
          new Rotation(el.inputStyle.get('--rotate')),
          new Scale(el.inputStyle.get('--scale-x'),
                    el.inputStyle.get('--scale-y'))
        ].concat(el.inputStyle.get('transform'))));
    }}, {
      inputProperties: ["--translate-x", "--translate-y",
                        "--scale-x", "--scale-y",
                        "--rotate", "transform"],
      outputProperties: ["transform"]
    }
});
</script>
